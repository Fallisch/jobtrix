## 3. Re-Test der neuen Verteidigungen

### RT1 — In-Memory Rate-Limit-Backstop

**Verteidigung:** `lib/rate-limit.ts:29-44` — `checkMemoryRateLimit()` mit `Map<string, MemoryEntry>`.

**Angriff 1: Bypass durch Key-Rotation.**
Hypothese: Angreifer wechselt IP-Adresse pro Request → jeder Request bekommt einen neuen Rate-Limit-Key → In-Memory-Store wird nie das Limit erreichen.
Analyse: Auth-Endpunkte (login, register, forgot/reset-password) nutzen `getClientIp()` als Key-Basis. Ein Angreifer mit vielen IPs (Botnet, VPN-Rotation) umgeht sowohl DB- als auch Memory-basiertes Rate-Limiting. `generate` und `send-email` nutzen `session.user.id` → Key-Rotation unmöglich.
**Verdikt:** ⚠️ bekannte Limitation bei IP-basiertem Rate-Limiting (Standard in der Branche). Auth-Endpunkte wären bei IP-Rotation angreifbar. Mitigiert durch: bcrypt-Kosten (~200ms pro Versuch verlangsamt Brute-Force natürlich), Account-Lockout existiert nicht → kein DoS-Risiko durch Sperren.

**Angriff 2: Memory-Exhaustion durch viele Keys.**
Hypothese: Angreifer sendet Requests mit tausenden verschiedenen Keys (verschiedene E-Mail-Adressen bei `/register`) → Map wächst unbegrenzt → OOM.
Analyse: `cleanupMemoryStore()` räumt alle 5 Minuten abgelaufene Einträge auf (`MEMORY_CLEANUP_INTERVAL_MS = 5 * 60 * 1000`). Window ist 15 Minuten. Ein Key verbraucht ~100 Bytes. Bei 10.000 verschiedenen IPs in 15 Minuten: ~1 MB. Unrealistisch, dass dies OOM verursacht.
**Verdikt:** ✅ gehalten — Cleanup-Mechanismus ausreichend, Memory-Footprint vernachlässigbar.

**Angriff 3: Race-Condition zwischen read und increment.**
Hypothese: Zwei parallele Requests mit gleichem Key lesen gleichzeitig `entry.attempts` → beide sehen 4 (unter Limit 5) → beide incrementieren → 6 statt Block.
Analyse: JavaScript ist single-threaded (Node.js Event Loop). `checkMemoryRateLimit()` ist synchron — kein `await` zwischen Read und Write. Race ist nicht möglich.
**Verdikt:** ✅ gehalten — synchrone Ausführung schließt Race aus.

### RT2 — Nonce-basierte CSP

**Verteidigung:** `middleware.ts:49-72` — generiert UUID-Nonce, setzt auf Request- und Response-Headers.

**Angriff 1: Nonce-Vorhersage.**
Hypothese: Angreifer errät den Nonce des nächsten Requests.
Analyse: Nonce wird mit `crypto.randomUUID()` generiert (128 Bit Entropie, kryptografisch sicher). Pro Request ein neuer Nonce. Nicht vorhersagbar.
**Verdikt:** ✅ gehalten.

**Angriff 2: Nonce-Leakage über Response-Headers.**
Hypothese: Angreifer liest den Nonce aus dem CSP-Header der Response und nutzt ihn für XSS.
Analyse: Der Nonce steht im CSP-Response-Header (`'nonce-...'`). Aber: der Angreifer bräuchte den Nonce *bevor* die Seite gerendert wird, um ein Script einzuschleusen. Da der Nonce pro Request generiert wird und die Response atomar ist, kann der Angreifer den Nonce einer laufenden Response nicht nutzen, um Code in *dieselbe* Response zu injizieren.
Cross-Origin-Request: CORS blockiert das Lesen von Response-Headers anderer Origins. Same-Origin: Wenn der Angreifer bereits JavaScript im Same-Origin-Kontext ausführt, hat er ohnehin vollen Zugriff.
**Verdikt:** ✅ gehalten.

**Angriff 3: CSP auf API-Routen fehlt.**
Hypothese: API-Routen haben keine CSP → JSON-Responses sind ungeschützt.
Analyse: Middleware-Matcher `["/((?!_next|_vercel|api|.*\\..*).*)"]` schließt API-Routen aus. API-Routen geben JSON zurück — CSP ist für JSON-Responses irrelevant (Browser wertet CSP nur für Dokument-Responses aus). Die Security-Headers (HSTS, nosniff, etc.) werden weiterhin über `next.config.mjs` für alle Routen gesetzt.
**Verdikt:** ✅ gehalten — korrekte Architektur.

**Angriff 4: Nonce wird nicht an next-themes weitergereicht.**
Analyse: `app/layout.tsx:10` liest Nonce aus `headers().get("x-nonce")`. `ThemeProvider` gibt `nonce` an `NextThemesProvider` weiter (`components/ThemeProvider.tsx:7`). next-themes 0.4.6 unterstützt die `nonce`-Prop und setzt sie auf sein Inline-Script.
**Verdikt:** ✅ gehalten — Nonce-Durchreichung korrekt implementiert.

### RT3 — JWT-Revocation nach Passwort-Reset

**Verteidigung:** `lib/auth.ts:73-77` — vergleicht `passwordChangedAt` mit `token.iat`.

**Angriff 1: Timing-Edge-Case.**
Hypothese: Password-Reset und Login passieren in derselben Sekunde → `changedAtSec === iat` → Check schlägt nicht zu → alter Token bleibt gültig.
Analyse: `changedAtSec > iat` (strict greater-than). Wenn beide in derselben Sekunde: `changedAtSec === iat` → Bedingung ist false → Token bleibt gültig. Das ist **korrekt**: ein Token, das in derselben Sekunde wie der Password-Reset ausgestellt wurde, repräsentiert die aktuelle Session (der User hat sich gerade erst eingeloggt und sofort das Passwort geändert).
**Verdikt:** ✅ gehalten — Edge-Case korrekt behandelt.

**Angriff 2: DB-Ausfall umgeht Revocation.**
Hypothese: Wenn `prisma.user.findUnique` im Session-Callback fehlschlägt, fällt der Code in den catch-Block → `session.user.id = token.id` wird gesetzt → gestohlener Token bleibt gültig trotz Passwort-Reset.
Analyse: `lib/auth.ts:88-92` — im catch-Block wird `session.user.id` gesetzt. Das ist beabsichtigt: DB-Ausfall soll den Nutzer nicht ausloggen (UX-Entscheidung aus der vorigen Endkontrolle). Bei einem DB-Ausfall ist auch der Login nicht möglich, was das Risikofenster begrenzt.
**Verdikt:** ⚠️ bekannte Limitation — bewusster Trade-off UX vs. Sicherheit. Bei DB-Ausfall greift die Revocation nicht. Akzeptables Restrisiko: der Angreifer müsste gleichzeitig den Token stehlen UND einen DB-Ausfall herbeiführen.

**Angriff 3: passwordChangedAt ist nullable → Altbestandsnutzer ohne Timestamp.**
Analyse: `user?.passwordChangedAt` — wenn null (Altbestandsnutzer, die ihr Passwort nie über Reset geändert haben), ist die Bedingung false → kein Check → Token bleibt gültig. Das ist **korrekt**: diese Nutzer haben ihr Passwort nie zurückgesetzt, also gibt es keinen Invalidierungsgrund.
**Verdikt:** ✅ gehalten — nullable ist der richtige Default.

### RT4 — Cron-Endpoint fail-closed + timingSafeEqual

**Verteidigung:** `app/api/cron/cleanup/route.ts:8-18`.

**Angriff 1: Leeres CRON_SECRET.**
Analyse: Zeile 9 — `if (!cronSecret) return 503`. Fail-closed. Kein Bypass möglich.
**Verdikt:** ✅ gehalten.

**Angriff 2: Timing-Attack auf den Secret-Vergleich.**
Analyse: Zeile 17 — `timingSafeEqual(authBuffer, expectedBuffer)` mit Length-Check davor. Korrekte Implementierung: unterschiedliche Längen werden sofort abgelehnt (leakt Längeninfo, aber Brute-Force auf Länge ist irrelevant bei UUID-Secrets). Gleiche Länge → konstante Vergleichszeit.
**Verdikt:** ✅ gehalten.

**Angriff 3: Buffer.from Encoding-Trick.**
Hypothese: `Buffer.from(authHeader)` interpretiert den Header als UTF-8. Könnte ein spezielles Encoding den Vergleich täuschen?
Analyse: Sowohl `authBuffer` als auch `expectedBuffer` nutzen `Buffer.from(string)` → UTF-8. Gleiche Codierung → kein Mismatch möglich. Der Angreifer kann keinen UTF-8-String senden, der byte-für-byte mit einem anderen UTF-8-String übereinstimmt aber semantisch anders ist.
**Verdikt:** ✅ gehalten.

### RT5 — Audit-Events für Passwort-Reset

**Verteidigung:** `forgot-password/route.ts:28`, `reset-password/route.ts:33`.

**Angriff: Audit-Log-Umgehung.**
Hypothese: Angreifer löst Passwort-Reset aus → Audit wird nicht geschrieben (z.B. DB-Fehler).
Analyse: `logAudit` in `lib/audit.ts` fängt DB-Fehler im catch → Audit-Failure bricht den Request nicht ab. Bei DB-Ausfall wird kein Audit geschrieben, aber der Reset geht trotzdem durch. Das ist der korrekte Trade-off: ein fehlgeschlagenes Audit darf den Nutzer nicht aus seinem Account aussperren.
**Verdikt:** ✅ gehalten — konsistentes Muster mit den anderen Audit-Stellen.
